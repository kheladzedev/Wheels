# Eval — `wheel_baseline_v1`

Source: `outputs/eval/wheel_baseline_v1.json` (generated by
`./scripts/eval_baseline.sh`).

- Model: `runs/pose/wheel_baseline_v1/weights/best.pt`
  (yolo11n-pose, 50 epochs, fitness 0.941).
- Dataset: `configs/pose_dataset.yaml` → `data/wheel_pose_dataset/val`
  (80 images, 44 ground-truth wheels).
- Inference: `--conf 0.25 --iou 0.45 --max-det 20 --device cpu`,
  OKS sigmas `(0.10, 0.10, 0.10)`.

## Caveat — what these numbers actually measure

`data/wheel_pose_dataset/val` is the converted output of
`data/incoming/real_v1/annotations/`, which is **396 auto-drafts**
produced by `auto_annotate_wheels.py` (COCO-vehicle + SAM-2 grid
prompts). The drafts are flagged `_draft: true,
_warning: NOT_GROUND_TRUTH_REQUIRES_HUMAN_REVIEW`.

So this report measures model agreement with *another model's
output*, not with human-verified labels. It pins the plumbing and
gives a sanity floor; it is **not** a generalisation signal. A
generalisation signal arrives after the QA pass through
`manual_keypoint_annotator.py --prefill-from …` (see
`docs/REAL_V1_RETRAIN.md`).

## Headline takeaway

Box detection works at acceptable mAP for a yolo11n run on 396 noisy
labels. Keypoint precision is roughly an order of magnitude better
than the 32-px tile this image is rendered at, but the high FP rate
(0.58) means the postprocessing layer / threshold sweep is the next
lever to pull *before* retraining on cleaner labels. The right next
step is the QA pass (see `docs/REAL_V1_RETRAIN.md`), not a deeper
model.
