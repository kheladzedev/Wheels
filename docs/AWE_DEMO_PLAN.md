# AWE Demo Plan — VSBL Wheel-Fitting ML/AR

Plan for a controlled, conference-style demo of the wheel-fitting
ML/AR mechanic. **Not a production-readiness claim, not an AR-ready
claim.** No retraining. No JSON contract changes.

## 1. Demo goal

Walk a viewer through the wheel-fitting pipeline in one minute:

1. ML detects every wheel in a single 2D camera frame.
2. ML emits a per-wheel bounding box and three named keypoints
   (`a`, `b`, `c_disc_bottom`) in screen coordinates.
3. AR takes those keypoints and reconstructs a vertical wheel plane
   plus the disc-bottom 3D position (this part is shown as a **mock
   visual explanation**, not as a live AR raycast).
4. The JSON contract between ML and AR is shown verbatim so the
   audience sees the boundary.

The viewer should leave the demo understanding **what ML emits, what
AR consumes, where 2D ends, and where 3D begins**.

## 3. What we cannot claim

The audience must not leave with any of these impressions, and the
narrator must not state them:

- "The model is production-ready" — production audit
  (`docs/PRODUCTION_READINESS_AUDIT.md`) is not green: export parity
  with real `WheelBBox` / `BBox` is unresolved
  (`docs/EXPORT_PARITY_AUDIT.md`), and the AR-device human-labelled
  holdout has not yet been collected.
- "The AR mechanic works end-to-end on device" — the mock plane
  rendered in the overlay is **drawn**, not raycast. No real
  raycast + RANSAC happens in this demo.
- "Keypoint accuracy meets the 5 px target" — current champion's
  median A / B / C pixel error sits at ~7.5 px on the real-only val
  split (`docs/AR_HANDOFF.md`).
- "TFLite / LiteRT is deployment-ready" — TFLite export exists but
  device validation against `validate_android_litert_report.py` has
  open items.
- "All wheels are detected" — false-negative rate on the real-only
  val split is 6.3 %, false-positive rate is 25.0 %
  (`outputs/eval/wheel_real_v1_self_plus_ue_synthetic_s_on_self_val.json`).
- "The synthetic cartoons prove anything about real-world quality" —
  they validate plumbing, not accuracy.

## 5. Fallback strategy

The demo must remain coherent if live inference is imperfect or weights
move. Layered fallbacks, in order of preference:

1. **Primary**: model-predicted overlays already on disk (`outputs/demo/`,
   `outputs/demo_self_s/`, `outputs/demo_soft_s_aug/`). Reused as
   `model_prediction`. No new inference required.
2. **Secondary**: auto-annotation drafts from
   `outputs/manual_real_auto_v3/` shown as `annotation_preview` with a
   visible "DRAFT — not human-verified" badge. Used to demonstrate the
   *concept* of A / B / C without claiming model accuracy.
3. **Tertiary**: synthetic-smoke overlays from
   `outputs/keypoint_preview/` and `outputs/pose_label_preview/train/`
   shown as `synthetic_smoke` with a "PLUMBING ONLY" badge. Used only
   to explain the *dataset format*, never to imply real-world quality.
4. **Universal**: mock AR visualisation (bbox + A/B/C + base line +
   semi-transparent vertical plane + disc-bottom marker) is generated
   from JSON regardless of the chosen primary tier, so the AR
   handoff story can be told even if every layer above fails. The
   plane is **drawn**, not raycast.

If the demo machine has no network and no GPU, layers 1–4 all run
from existing artefacts on disk.

## 6. Demo script / storyline

Suggested two-minute walkthrough. Narration cues in *italics*.

1. **Title slide** — *"AR wheel fitting — how the ML side hands off to
   AR."* Open `outputs/awe_demo/report/contact_sheet.jpg`.
2. **Real input** — pick one model_prediction frame. *"This is a real
   camera frame. The ML model runs once per frame, single class, three
   named keypoints per detected wheel."*
3. **Model output** — overlay bbox + A / B / C from the same frame.
   *"A and B are screen-space points the AR side will raycast onto the
   floor. C is the lowest visible point of the metal rim."*
4. **JSON contract** — open the matching `outputs/awe_demo/json/*.json`.
   *"This is the entire contract. No 3D, no track ID, no timestamp,
   no per-keypoint confidence. Pixel coordinates only."*
5. **AR mock** — open the matching `__ar_mock.jpg`. *"On the AR side,
   A and B get raycast onto the floor. The line between their world-
   space hits is the base of a vertical plane. C lands on that plane
   and gives us the disc-bottom 3D position. This image is a mock
   visualisation, not a live raycast — the real raycast and RANSAC run
   on the device."*
6. **JSON ↔ Mock pairing** — flip back and forth between the JSON
   pane and the AR-mock overlay to show that each line in the JSON
   corresponds to one drawn element.
7. **Limits** — pick one frame with a known false positive or large
   error. *"Median A / B / C pixel error is around 7.5 px on the real-
   only val split, and we have outstanding work on export parity and
   human-labelled holdout data before we call this production-ready."*
8. **Close** — *"ML stays 2D, AR owns 3D. The boundary is the JSON on
   screen. Questions?"*

## 7. Risks

| Risk | Mitigation |
|---|---|
| Audience walks away thinking the model is production-grade | Narration plus on-overlay badges plus `outputs/awe_demo/README.md` "what to say / what not to say" section. |
| Mock AR visualisation mistaken for a live AR feed | Persistent "AR MOCK — visual explanation, not real raycast" badge in the bottom-left of every mock frame. |
| Cherry-picked frames hide failure modes | Curated set includes at least one frame with a low-confidence wheel or known issue, narrated in step 7 above. |
| Schema drift between demo and what AR consumes | Demo JSON is copied byte-for-byte from `outputs/demo/json/`; the same files that `tests/test_ar_contract.py` pin. No reformatting. |
| Demo machine can't run inference | All artefacts are pre-rendered. No model loaded at demo time. |
| Synthetic-smoke frame shown without label | Build script writes the provenance badge into the JPEG and into `demo_summary.json`; the README spells out the labels. |
| Demo overlays leaked into production audit reports | Demo lives under `outputs/awe_demo/`; production gates read from `outputs/production_audit/` and `outputs/eval/`. No code path mixes them. |

## 8. Next steps after demo

The demo is not a substitute for the production track. After the
showcase, the work queue stays:

1. Close the export blocker: clean `WheelBBox` / `BBox` from the
   exporter, re-run `src/export_certification.py` and
   `src/export_parity_audit.py` to acceptance
   (`docs/EXPORT_CERTIFICATION.md`).
2. Capture a human-labelled AR-device holdout (`docs/AR_ML_CONTRACT.md`
   §"Production Holdout Provenance") and run
   `src/evaluate_ar_holdout.py` against it.
3. Record a real-device AR replay log and run
   `src/validate_ar_replay.py`; once Stage 3 of
   `docs/AR_REPLAY_METRIC_PLAN.md` is implemented, score it.
4. Re-train against any human-corrected drafts surfaced by the demo
   review — only after step 1, never on the demo-only data.

## 9. Non-goals (load-bearing)

- **Not** a new model. No training, no fine-tuning, no
  hyperparameter sweeps for the demo.
- **Not** a new JSON. The demo emits the same confirmed schema
  pinned by `tests/test_ar_contract.py` and
  `tests/test_confirmed_ar_schema_shape.py`.
- **Not** a deletion of existing artefacts. The build script writes
  only into `outputs/awe_demo/`; nothing under `outputs/demo*/`,
  `outputs/eval/`, `outputs/production_audit/`, `runs/`, or `data/`
  is modified or removed.
- **Not** a tracking demo. No persistent IDs across frames, no
  fake video tracking.
- **Not** AR-ready certification. Production gates remain unchanged.
