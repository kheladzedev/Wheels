# Нейросеть «Поиск колёс» — спецификация (ревизия)

Исправленная версия присланной спеки. Правки приведены в соответствие с
уже зафиксированным контрактом ML↔AR и геометрией. Источники истины:
`docs/KEYPOINT_SPEC.md`, `docs/AR_ML_CONTRACT.md`,
`docs/OPEN_QUESTIONS_AR_SPEC.md`, `docs/MODEL_ARCHITECTURE_PROPOSAL.md`,
`docs/ANDROID_FIRST_MODEL_PLAN.md`, `docs/SYNTHETIC_DATA_STATUS.md`,
реализация `src/models/*`.

## Что изменено относительно присланной версии

1. **A/B — это floor-ray точки на полу у пятна контакта, НЕ точки на
   колесе.** Имена приведены к контрактным `a` / `b` / `c_disc_bottom`.
   Старая семантика «нижняя граница колеса / край обода» запрещена с
   2026-05-14 (`KEYPOINT_SPEC.md:13-22`).
2. **C raycast пересекается с восстановленной ВЕРТИКАЛЬНОЙ плоскостью
   колеса, не с полом.** Зафиксирован порядок: A/B → пол → K кадров +
   RANSAC → вертикальная плоскость → C → эта плоскость.
3. **Intrinsics (focal/FOV) — обязательный вход 3D-реконструкции.**
   `{pitch, roll, deltaZ}` это поза плоскости (extrinsic), не intrinsics.
4. **`floor_input` убран из mobile.** Mobile-ML по контракту stateless и
   2D-only. Подача пола на вход — отдельный эксперимент с новым
   контрактом, не текущий путь.
5. **web `deltaZ`: честно помечена монокулярная масштабная
   неоднозначность.** Нужен метрический якорь/приор либо
   scale-relative выход.
6. **Данные: реальные ≥2000 + hard negatives + domain randomization;
   сначала фикс UE-плагина и переразметка под floor-ray.**
7. **Метрика приёмки — 3D (disc-height sigma), split по `scene_id`.**

Сопутствующие правки, без которых спека внутренне противоречива,
помечены ниже как `[fix]`.

# Этап 1: сбор данных и первичное обучение

## Источники данных

Источники: реальные AR-session кадры (камера телефона, 1–3 м от авто,
помещение + улица) как **основной** источник, плюс сцены в UE5 для
претрейна/аугментации и ручная 2D-разметка.

`[fix-6]` UE5 **не** ставится единственным основным источником. Эмпирика
репо: UE/синтетика-only деградирует на реале (фары/зеркала срабатывают
как колёса; UE-only OKS ≈ 0.15 против mixed-real чемпиона OKS ≈ 0.89,
`SYNTHETIC_DATA_STATUS.md`). Для production обязательны:

| Бакет | Цель | Назначение |
|---|---:|---|
| Реальные AR-session-like | ≥ 2000 | основной обучающий сигнал |
| Реальные occluded-кадры | ≥ 200 | негативы (не эмитятся, контракт §3) |
| Реальные no-car | ≥ 200 | hard negatives: полы, стены, свет |
| Реальные wrong-target | ≥ 200 | велосипеды/мотоциклы — круглое, но НЕ авто |
| Реальные сложный фон | ≥ 100 | мокрый асфальт, блики, грязные диски, тень |
| Синтетика UE5/cartoon | ≤ 10% | претрейн/доменное расширение, малая доля |

Плюс domain randomization по свету, текстурам, фонам, ракурсам.

`[fix-6]` **Готовность UE-выгрузки.** Текущий плагин к обучению не готов:
батч 0003 дал usable только 7.17% (24005 слотов → 1722 колеса, остальное
all-zero/out-of-bounds). Валидная UE-синтетика = 51 кадр / 90 колёс и
помечена draft, потому что A/B получены 5-to-3 эвристикой из rim-точек —
это запрещённая rim-производная семантика. Перед использованием UE как
источника нужно: (а) фикс плагина (не писать `.txt` для невидимых/вне
кадра колёс, экспортировать `bbox_xyxy`, `scene_id`); (б) переразметка
A/B под floor-ray контракт.

## Разметка keypoints

Для каждого **полностью видимого** колеса размечаются ровно 3 точки в 2D
(screen-space, пиксели, origin top-left):

```
wheel = {
  points: {
    a:             [x, y],   // левая floor-ray точка
    b:             [x, y],   // правая floor-ray точка
    c_disc_bottom: [x, y]    // нижняя видимая точка металлического диска
  }
}
```

`[fix-1]` Семантика точек (зафиксирована AR-командой 2026-05-13,
уточнена 2026-05-14, переподтверждена 2026-05-18):

- `a`, `b` — **точки на полу рядом с пятном контакта колеса**
  (floor-ray points). AR пускает из них луч в плоскость пола; две
  floor-проекции задают базовую линию вертикальной плоскости колеса.
  **Не на покрышке и не на ободе.** Их определяющее свойство — лежать на
  плоскости пола, чтобы пересечение луча было точным.
- `c_disc_bottom` — **нижняя видимая точка металлического диска/обода**
  (не покрышка, не центр ступицы).

Геометрическое обоснование `a`/`b` на полу: луч восстанавливает истинную
3D-точку, только если пиксель лежит на целевой плоскости. Нижний видимый
край покрышки физически выше пятна контакта на высоту `h`, его raycast в
пол даёт **горизонтальное** смещение вдоль пола `e ≈ h / tan(θ)` (θ — угол
луча к плоскости пола): точка уезжает по полу, а не по высоте, то есть
ошибка попадает прямо в горизонтальный базис плоскости, который и фитит
RANSAC. На скользящих углах телефона у авто (θ малый) ошибка растёт
неограниченно (h=3 см, θ=10° → ~17 см).

`[fix]` **Occlusion.** Частично перекрытые колёса в разметку и в выход
**не попадают**. Флага visibility в выходном контракте нет. Внутренние
YOLO-pose метки несут `v ∈ {0,1,2}` для лоссов, но в AR-выход уходят
только полностью видимые колёса (`KEYPOINT_SPEC.md`, `DATASET_SPEC.md`).

## Данные о камере и поле (для web и для этапа 2)

В UE5 дополнительно собираются:

- поза камеры относительно плоскости пола: `pitch`, `roll`, `deltaZ`;
- `[fix-3]` **intrinsics: `FOV` (и/или focal length)**. Принципиальную
  точку для UE-pinhole берём как центр кадра `(W/2, H/2)`. Из FOV
  получаем `fx = (W/2) / tan(FOV_x / 2)`. Без intrinsics операция
  `пиксель → луч` (`dir = K⁻¹·[u,v,1]`) не определена, и raycast/3D-loss
  построить нельзя.

`[fix]` Обогащение разметки floor-плоскостью через существующую сеть
(ковры/напольные покрытия) допускается как bootstrap, но это
pseudo-label с шумовым потолком — валидировать против UE ground-truth.

## Мобильная версия

### Вход

`[fix]` Изображение **384×384 или 416×416** (кратно output stride 32:
`12×32=384`, `13×32=416`). Базовый ориентир — меньший из них из-за
производительности. **390×390 не использовать**: 390/32 = 12.19 — не
кратно. Padded stride-2 цепочка backbone даёт `390→195→98→49→25→13`
(`floor((n+2p−k)/s)+1`), т.е. сетку **13×13 — ровно как у входа 416**. Но
decode/matcher строят центры ячеек как `(i+0.5)·32` и неявно считают
`input = grid·32 = 416`; если в сеть подать настоящий canvas 390, кейпоинты
декодируются в системе координат 416 → рассинхрон рамки ~26 px у
правого/нижнего края.

`[fix-C]` Реализация в репо (`MODEL_ARCHITECTURE_PROPOSAL.md §2.2`,
`mobilenetv2_skipless_pose.py`) сейчас гоняет 640×640. Размеры 384/416 —
это перф-цель Игоря, её надо валидировать замером on-device latency;
decode/matcher параметризованы по stride, так что работают на любом
кратном 32. Точный размер выбрать после замера.

### Выход

Массив колёс. Для каждого — `points: {a, b, c_disc_bottom}` в 2D
(см. контракт выше). Только 2D-пиксели.

`[fix]` **Несколько колёс на кадре.** Encoder сам отдаёт одну feature
map; список инстансов рождается головой детекции + decode + порог по
`cls` + NMS. Голова — anchor-free FCOS-style, decoupled `cls/bbox/kpt`,
один инстанс на ячейку (реализация: `src/models/mobilenetv2_skipless_pose.py`,
`src/models/matcher.py`).

### Важные ограничения

Инстансы и ID колёс между кадрами **не согласованы**. Сопоставление —
позже, на этапе AR-фильтрации (AR кластеризует по 3D-позиции после
raycast; `track_id` из ML не идёт).

### Архитектура

`[fix]` Encoder MobileNetV2, **single-scale, без FPN** (используется
только финальная C5-фича stride-32). Формулировку «без самого нижнего
skip-connection» не используем: в ванильном MobileNetV2 FPN-скипов нет, а
внутренние residual-shortcut'ы inverted-residual блоков удалять нельзя
(сломает backbone и ImageNet-transfer). Конфигурация проверена по
производительности в real-time.

`[fix]` **Субпиксельная точность.** Сетка stride-32 даёт ~32 px
квантование, что ломает AR-RANSAC. Кейпоинты регрессируются как
center-offset `(dx, dy)` в долях ячейки → субпиксельно. При нехватке
точности — refinement-голова на stride-16/8.

Решение не обязано работать на каждом кадре в real-time; при заметном
приросте качества допустимы другие архитектуры.

`[fix]` **Бюджеты под вопросом.** До имплементации production нужно
зафиксировать: целевой Android-tier/чипсет, бюджет latency на кадр, cap
размера `.tflite`, допустимую просадку INT8 (`MODEL_ARCHITECTURE_PROPOSAL.md §7`).

## Web-версия

### Вход

Изображение до 512×512 (512 = 16×32, кратно stride).

### Выход

1. Параметры плоскости пола `floor = {pitch, roll, deltaZ}`.
2. Массив колёс с `points: {a, b, c_disc_bottom}` в 2D.

Параметризация плоскости:

```
floor = {
  pitch,    // наклон камеры относительно пола (угол)
  roll,     // наклон камеры относительно пола (угол)
  deltaZ    // перпендикулярное расстояние камера → плоскость пола
}
```

`[fix-5]` Это корректная минимальная параметризация плоскости (3 DOF:
нормаль `pitch`+`roll` = 2 DOF, расстояние `deltaZ` = 1 DOF; yaw
правильно опущен — поворот камеры вокруг нормали пола не меняет ни
нормаль, ни расстояние до плоскости).

**Важная оговорка по `deltaZ`:** углы `pitch`/`roll` безразмерны и
наблюдаемы из одного RGB. Абсолютный метрический `deltaZ` из
монокулярного pinhole-кадра **не восстановим без масштабного якоря**
(depth-vs-size неоднозначность). Известный FOV это **не лечит** — он
фиксирует только углы. Варианты:

- завести метрический якорь/приор (например, приор на радиус колеса) и
  предсказывать `deltaZ` метрически на его основе (хрупко на
  нестандартных авто/кропах — отслеживать failure-моды);
- либо отдавать scale-relative геометрию и резолвить масштаб downstream.

`[fix-3]` На web на инференсе UE-FOV отсутствует. Решить: предсказывать
focal/FOV сетью или фиксировать FOV приложения. Фиксированный FOV
усугубляет неоднозначность `deltaZ` — учитывать.

`[fix]` Web floor-head — **новое направление**, его нет в текущем
mobile-контракте (там ML строго 2D-only, плоскость владеет AR). Для web
нужен собственный контракт на выход floor-параметров.

### Архитектура

Свободнее: YOLO-подобная или multi-task (детекция + keypoints + оценка
плоскости пола). Реконструкцию параметров пола делать из глубоких
глобальных фич (bottleneck / GAP над C5) — пол это низкочастотное
глобальное свойство кадра; гипотеза ранее проверялась и работала.

# Этап 2: обучение с 3D-реконструкцией

## Общая логика (mobile и web)

`[fix-1][fix-2][fix-3]` Точная цепочка:

1. Сеть предсказывает 2D keypoints `a`, `b`, `c_disc_bottom`.
2. По intrinsics строится луч из пикселя: `dir = K⁻¹·[u, v, 1]`.
3. **`a` и `b` raycast-ятся на ПЛОСКОСТЬ ПОЛА** → две floor-проекции →
   накопление по K кадрам + RANSAC → **вертикальная плоскость колеса**
   (перпендикулярна полу, проходит через базовую линию a/b).
4. **`c_disc_bottom` raycast-ится на УЖЕ ВОССТАНОВЛЕННУЮ ВЕРТИКАЛЬНУЮ
   ПЛОСКОСТЬ (не на пол)** → 3D-позиция низа диска; усреднение по K
   валидным кадрам → высота установки диска. C физически выше пола, её
   нельзя пускать в пол — иначе получится точка опоры, а не высота диска.
5. На обучении 2D-предсказания дифференцируемо переводятся в 3D по этой
   же цепочке.
6. Loss считается и по 2D-координатам, и по ошибке в 3D-точках.

`[fix-3]` **Вход 3D-реконструкции включает intrinsics** (focal/FOV +
принципиальная точка = центр кадра). Без них шаг 2 не определён.

`[fix]` **Источник 3D ground-truth.** 3D-loss требует 3D-референса
(истинные 3D-позиции floor-проекций a/b и точки C, либо истинная
плоскость + intrinsics/extrinsics). Такой сигнал существует только в
UE-домене (`Goal/*` + геометрия мешей). На реальных данных 3D-GT нет →
3D-loss обучаем на синтетике; это снова упирается в sim-to-real, учитывать
при планировании.

`[fix]` **Численная стабильность.** Пересечение `t = -(n·o + d)/(n·dir)`
уходит в бесконечность при `n·dir → 0`, т.е. когда луч `a`/`b` идёт почти
параллельно полу (grazing incidence): малый угол луча к плоскости пола —
низкая камера, пологий ракурс, точки у линии горизонта. Якобиан точки
пересечения по 2D-пикселю масштабируется как ~`1/sin²(α)` (α — угол луча
к полу). Использовать robust-loss (Huber) + clamp дистанции пересечения;
возможно down-weight near-grazing лучей.

## Мобильная версия

`[fix-4]` Плоскость пола **не подаётся на вход** мобильной сети. Mobile-ML
по контракту per-frame, stateless, 2D-only (`AR_ML_CONTRACT.md:38-49`,
закреплено `tests/test_ar_contract.py`). Это согласуется с «Итогом»:
плоскость берётся из AR после инференса.

Эксперимент с `floor_input = [pitch, roll, deltaZ]` (нормализованным) как
дополнительным входом допустим **только как отдельный новый контракт**, не
как текущий путь. Риски, если делать: зависимость от оценки пола из AR
(на старте сессии она самая шумная, ARKit/ARCore сходятся за секунды) и
рассинхрон train/inference-пола. На обучении floor подаётся как
фиксированный вход (сеть его не предсказывает).

## Web-версия

Аналогично, но плоскость **предсказывает сама сеть**, затем keypoints и
предсказанная плоскость идут в 3D-реконструкцию (raycast против
собственной предсказанной плоскости).

### Балансировка multi-task (staged training)

Риск: на ранних этапах ошибки пола портят keypoint-обучение через
геометрическую связь (особенно масштаб через `deltaZ`:
`t = deltaZ / (dir·n_floor)`). Порядок:

1. стабилизировать 2D keypoints (прямые 2D-метки);
2. отдельно стабилизировать пол `{pitch, roll, deltaZ}` (UE GT);
3. постепенно подключать 3D reconstruction loss;
4. совместный fine-tune всей multi-task.

`[fix]` Усиления к staged training:

- **stop-gradient / detach на пол в 3D-loss**, пока пол не стабилизируется
  (чтобы плохой пол не тёк градиентом в keypoints);
- **ramp веса** 3D-члена от 0;
- балансировка разнородных целей (радианы / px / глубина): uncertainty
  weighting (Kendall) или GradNorm, а не только фиксированные λ.

Сейчас `src/models/loss.py` чисто 2D (focal+GIoU+SmoothL1+BCE); 3D-член —
новая работа. На далёких floor-ray `a`/`b` OKS с малой sigma насыщается —
поэтому для keypoints используется SmoothL1 (учесть при добавлении 3D).

# Итоговая цель

## Мобильная версия

Сеть предсказывает только 2D keypoints (`a`, `b`, `c_disc_bottom`).
Плоскость пола берётся из AR. 3D-реконструкция — после нескольких
инференсов, через raycast (`a`/`b` → пол → RANSAC → вертикальная
плоскость; `c_disc_bottom` → эта плоскость) и фильтрацию в AR-механике.

## Web-версия

Сеть предсказывает одновременно плоскость пола и 2D keypoints. Web
использует предсказанную плоскость + keypoints для 3D-реконструкции (с
оговоркой про масштаб `deltaZ`).

# Метрики приёмки

`[fix-7]` Качество меряется **в 3D**, не по 2D-пиксельной ошибке. AR-команда
дала только принцип: бюджет «по-хорошему надо считать в 3D»
(`OPEN_QUESTIONS_AR_SPEC.md §9`), конкретный AR-side бюджет ещё **не
подтверждён**. Конкретные пороги ниже — ML-side предложение
(`MODEL_ARCHITECTURE_PROPOSAL.md §4`), которое надо согласовать с AR:

- **disc-height sigma (K=10 RANSAC): < 3 см (v1) / < 1 см (prod)** —
  главная пользовательски значимая метрика (ориентир, pending AR).
- FP-rate на hard negatives: < 1.0 ложное колесо/кадр (v1) → < 0.1
  (prod).
- 2D px-ошибка / pose mAP — вспомогательный сигнал, не критерий приёмки.

`[fix-7]` **Split по `scene_id`.** UE-ран = ~100 коррелированных кадров с
общей траекторией камеры. Делить train/val **по сценам**, не покадрово,
иначе утечка коррелированных кадров → оптимистичные метрики. Нужен
`scene_id` из плагина; дефолтный покадровый shuffle-split — scene-blind.

# Ключевые требования

1. Несколько колёс на кадре (детекция-голова + decode + NMS).
2. На колесо — 3 keypoints: `a`, `b` (floor-ray, на полу),
   `c_disc_bottom` (низ диска).
3. Сопоставление колёс между кадрами — на этапе AR-фильтрации.
4. Mobile — лёгкая MobileNetV2-like (single-scale, без FPN), 2D-only,
   stateless; вход 384/416, кратный stride 32.
5. Web — допустима более тяжёлая multi-task; floor-head — новый контракт.
6. Этап 2 — 3D reconstruction loss; обязателен вход с intrinsics и
   источник 3D-GT; raycast `a`/`b` на пол, `c_disc_bottom` на
   восстановленную вертикальную плоскость.
7. web multi-task балансировать осторожно (staged + detach пола + ramp +
   uncertainty/GradNorm).
8. Приёмка — в 3D (disc-height sigma); split по `scene_id`.
