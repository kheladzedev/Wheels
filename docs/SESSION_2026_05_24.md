## What worked

### 1. Headless `--init-from` fine-tune (the day's main win)

`scripts/train_mobilenetv2_skipless.py` now accepts `--init-from CHECKPOINT`
that warm-starts `MobileNetV2SkiplessPose` from a saved `.pt` before training.
Critical fix: the project's checkpoints serialise the model weights under the
key `model_state_dict`; the loader now tries `model_state_dict`, `model`, and
`state_dict` in that order. Verified by a clean training run where epoch 1
step 1 train loss was `0.7253` (close to the prev val 0.640) instead of the
random-init `8.86` we saw without the fix.

Two fine-tune runs landed:

| Run | Init from | Added data | Val loss | 0003 smoke (8 frames) |
|---|---|---:|---:|---:|
| `mn2_finetune_prevv3_on_diversified_e5_v3` | prev v3 | 1000 standartWheelsRoom (diversified) | 0.640 | **6 confirmed** |
| `mn2_finetune_prevv3_on_wikimedia_v2_e5` | prev v3 | 65 Wikimedia photos auto-annotated (21 wheels kept) | 0.747 | **6 confirmed** |

Both match the prev v3 baseline on the 0003 smoke. Neither makes a measurably
better model on this 8-frame probe, but importantly **neither regresses**.
That is a real change from yesterday, when every vanilla retrain attempt
collapsed the model to 2 confirmed wheels.

**editor mode (not commandlet)** via the binary flag
`-ExecutePythonScript=/abs/path/to/script.py`. The same Python that crashed in

with `unreal.SystemLibrary.quit_editor()` so the shell command returned
without a human touching the GUI.

A 1000-frame capture loop on the new map ran end-to-end. Frames now show

### 5. Wikimedia query mix still wastes most of the candidates

Built-in query mix in `src/fetch_wikimedia_cars.py` returned 1737 candidates
to download 52 new images. The Wikimedia matches are dominated by
parking-lot architecture, aircraft, monuments, and people movers. To get a
real-world batch larger than ~50–80 cars we need a narrower query list
(e.g. car-specific make/model names, "sedan side parked", "hatchback rear
quarter") and a domain-aware filter that drops images where the COCO
detector finds no vehicle bbox at all.

## Native-bbox augmentation outcome

`scripts/augment_native_bbox.py` (COCO+SAM2 → existing-point match → replace
derived bbox with mask-derived bbox) ran over all 1000 multivehicle frames.

Result: 37 of 307 existing wheels (12%) got a native bbox. The remaining
259 wheels had no detector match (small / shadowed / partial wheels) and 11
were rejected for height mismatch. After re-converting and fine-tuning from
prev v3, the 0003 smoke gave **5/10 confirmed wheels**, identical to the
non-augmented multivehicle fine-tune. 12% replacement coverage is too small
to shift the model.

To actually beat the 6/10 baseline we need one of:

- **UE-side native wheel mesh projection.** Use the 62 separate
  `SM_Wheel_*` components: for each vehicle in the capture frame, project
  each wheel mesh's world-space bounds into screen space and write the
  resulting `BBox` line. This is a build-time operation in UE Python and
  should yield >80% coverage on visible wheels. Not done today because the
  capture-pipeline modification is non-trivial and Igor owns the exporter
  graph.
- **Human-QA real photos.** `src/manual_keypoint_annotator.py` lets the
  operator click through `prefill-from` auto-drafts and confirm A/B/C
  per wheel. Bottlenecked on a human session, not on engineering.
- **Igor delivers native `BBox`/`WheelBBox` in the exporter.** This is the
  preferred path — same data, simpler intake.

## Engineering hygiene additions to memory

- `pkill -f` ERE vs BRE alternation — `pattern_a|pattern_b` not `\|`.
  (Already in `memory/feedback_pkill_regex.md`.)
- UE `-pythonscript` headless `spawn_actor_from_object()` on a World
  Partition-aware map crashes with "Caught signal" even under
  `-AllowCommandletRendering`. Stick to mutating existing actors and
  `save_current_level()` in commandlet mode; for net-new actor placement
  use the editor GUI.
- Map 03 / 02 / 01 in this project rely on World Partition spatial cells.
  In headless `-pythonscript` with `NullRHI` and in `-game` mode the cells
  never load, so the maps report zero `wheels`/`CameraCaptureWheels`
  actors. Until Igor either disables WP for these or hands over an
  explicit cell-activation snippet, only `standartWheelsRoom*` is usable
  for capture from this side.
- `bash -c '... | head -N'` background launches send SIGPIPE to the child
  after N lines, which kills long-running scripts. Always redirect to a
  log file with `> /tmp/foo.log 2>&1` for background trainers.
- macOS hot corners and Notification Center steal clicks near the top-right
  edge from computer-use, even with the app granted. Pull windows away from
  the edges before automating menus.
